# Research notes - 2026-04-08

## Assignment

Build an open source, agent-first Rust CLI for short-form video composition.

Immediate questions:

1. How can we pull trending sounds from TikTok programmatically?
2. How can we pull viral video clips from Twitter/X?
3. How do we combine those into short clips suitable for posting on Twitter/X?
4. What is the right agent-first CLI architecture?

## TikTok trending sounds

### Early read

There does not appear to be a straightforward official TikTok API surface specifically for trending sounds discovery in the way we want.

Promising paths found:

- Unofficial TikTok wrappers and scraping-oriented libraries such as `TikTokApi`
- Managed unofficial APIs such as TikAPI
- Trend-focused third-party services such as trendsmcp/tiktok-trends-api
- Commercial scraping products like Apify actors focused on TikTok music trends

### Practical conclusion

For an MVP, TikTok sound discovery will likely need one of these modes:

1. **Provider mode** using a managed unofficial API
2. **Scraper mode** using an unofficial library or browser automation
3. **Import mode** where agents ingest known trending sound URLs/IDs and normalize them into the library

The safest short-term engineering path is probably:

- start with provider adapters
- support imported sound metadata
- keep direct scraping behind an optional adapter boundary

### Risks

- instability from unofficial APIs
- anti-bot measures
- changing HTML/app contracts
- unclear rights around downloading and reusing audio

## Twitter/X viral clips

### Early read

X API docs suggest a usable route for discovery:

- recent search endpoints for posts
- filtering for posts with media/video
- collecting engagement metrics from post/media expansions
- ranking by public metrics and recency

### Practical conclusion

An MVP can likely define “viral” operationally rather than mystically.

For example:

- search for posts matching a topic query
- require attached video/media
- fetch public metrics
- compute a score from likes, reposts, replies, quotes, views, and recency
- select top candidates

Video extraction/downloading may still need non-official handling depending on what the X API exposes for actual media retrieval in practice.

So the likely split is:

- official API for search and ranking
- separate downloader/import adapter for the media asset itself

### Risks

- X API access tiers and rate limits
- media download friction
- copyright/reuse issues

## MVP architecture

### Proposed layers

1. **research/**
   - source adapters for TikTok and X
   - ranking and discovery primitives

2. **library/**
   - local manifests for sounds and clips
   - provenance, tags, durations, hashes, source URLs

3. **media/**
   - ffmpeg wrappers and filtergraph generation
   - trim, crop, normalize loudness, caption burn-in, mux

4. **compose/**
   - choose sound + clips
   - build assembly plan
   - render final short

5. **cli/**
   - agent-facing commands and JSON output

### Agent-first primitives

The CLI should expose small verbs instead of giant magical workflows.

Examples:

- `capcut-cli discover sounds tiktok`
- `capcut-cli discover clips x --query <topic>`
- `capcut-cli library add sound --from <source>`
- `capcut-cli library add clip --from <source>`
- `capcut-cli compose short --sound <id> --clip <id> --clip <id>`
- `capcut-cli render plan.json`

## Likely system dependencies

- Rust CLI core
- ffmpeg installed locally
- optional sqlite database for metadata/indexing
- optional provider credentials for X and third-party TikTok data services
