# MVP plan - 2026-04-08

> **Status (2026-04-12):** All items below are implemented. The full pipeline
> (discover → import → compose → MP4) is working end-to-end.

## First execution path

Build the CLI in layers, but bias the first working path toward the easiest credible prototype:

1. Discover viral X/Twitter video posts for a topic
2. Import or attach a trending TikTok sound via provider/import adapter
3. Generate a composition plan
4. Render a short clip via ffmpeg

## Why this order

- X discovery has a clearer official search surface than TikTok sound discovery
- TikTok sounds are likely to require adapter boundaries anyway
- A composition plan can be useful before full rendering is automatic
- Agent users need inspectable JSON before they need magic

## Initial command contract

### Discovery

- `capcut-cli discover tiktok-sounds --limit 10`
- `capcut-cli discover x-clips --query "ai agents" --limit 10`

### Library planning

- `capcut-cli library sound --from <url-or-provider> --id <optional-id>`
- `capcut-cli library clip --from <url-or-provider> --id <optional-id>`

### Composition planning

- `capcut-cli compose --sound sound_123 --clip clip_a --clip clip_b --duration-seconds 30`

## Implementation modules

- `src/cli.rs` — command tree and JSON reports
- `src/models.rs` — report schemas and asset types
- `src/library.rs` — on-disk manifest and asset CRUD
- `src/downloader.rs` — yt-dlp integration (probe + download)
- `src/ffmpeg.rs` — render pipeline (scale, crop, normalize, mux)

## Dependencies

- `clap` — command structure (derive macros)
- `serde` + `serde_json` — machine-readable JSON output
- `anyhow` — error handling
- `uuid` — asset ID generation
- `dirs` — platform data directory
- External: `yt-dlp` (media download), `ffmpeg` (render pipeline)

## Research constraints to keep visible

- TikTok trending sounds are the least stable acquisition surface
- X search may be official, but media retrieval may not be
- audio and clip reuse rights need provenance and explicit notes
